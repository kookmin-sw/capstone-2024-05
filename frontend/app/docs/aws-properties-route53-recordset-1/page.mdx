# AWS::Route53::RecordSet그룹 레코드 세트

만들려는 하나의 레코드에 대한 정보입니다.

## 통사론

AWS CloudFormation 템플릿에서 이 엔터티를 선언하려면 다음 구문을 사용합니다.

### JSON (영문)

```
{
  "AliasTarget" : AliasTarget,
  "CidrRoutingConfig" : CidrRoutingConfig,
  "Failover" : String,
  "GeoLocation" : GeoLocation,
  "HealthCheckId" : String,
  "HostedZoneId" : String,
  "HostedZoneName" : String,
  "MultiValueAnswer" : Boolean,
  "Name" : String,
  "Region" : String,
  "ResourceRecords" : [ String, ... ],
  "SetIdentifier" : String,
  "TTL" : String,
  "Type" : String,
  "Weight" : Integer
}
```

## 속성

`AliasTarget`  
 _별칭 리소스 레코드 세트만:_ 트래픽을 라우팅하려는 AWS 리소스(예: CloudFront 배포 또는 Amazon S3 버킷)에 대한 정보입니다.  
프라이빗 호스팅 영역에 대한 리소스 레코드 세트를 생성하는 경우 다음 사항에 유의하세요.

- 트래픽을 CloudFront 배포로 라우팅하기 위해 프라이빗 호스팅 영역에 별칭 리소스 레코드 세트를 생성할 수 없습니다.
- 프라이빗 호스팅 영역에서 장애 조치 리소스 레코드 세트를 생성하는 방법에 대한 자세한 내용은 _Amazon Route 53 개발자 안내서의 프라이빗 호스팅 영역에서 장애 조치 구성을 참조하십시오_.
  *필수 항목 *여부: 아니요  
  _유형_: AliasTarget  
  _업데이트 필요 사항_: 중단 없음

`CidrRoutingConfig`  
속성 설명을 사용할 수 없습니다.  
*필수 항목 *여부: 아니요  
_유형_: CidrRoutingConfig  
_업데이트 필요 사항_: 중단 없음

`Failover`  
 _장애 조치 리소스 레코드 세트만:_ 장애 조치를 구성하려면 `Failover` 두 개의 리소스 레코드 세트에 요소를 추가합니다. 한 리소스 레코드 세트 의 `PRIMARY` 경우 의 값으로 `Failover` 지정하고, 다른 리소스 레코드 세트의 경우 `SECONDARY`를 지정합니다. 또한, 요소를 포함 `HealthCheckId` 시키고 Amazon Route 53이 각 리소스 레코드 세트에 대해 수행할 상태 확인을 지정합니다.  
명시된 경우를 제외하고, 다음 장애 조치 동작은 `HealthCheckId` 두 리소스 레코드 세트 모두에 요소를 포함시켰다고 가정합니다.

- 기본 리소스 레코드 세트가 정상일 때, Route 53은 보조 리소스 레코드 세트의 상태와 관계없이 기본 리소스 레코드 세트의 해당 값으로 DNS 쿼리에 응답합니다.
- 기본 리소스 레코드 세트가 비정상이고 보조 리소스 레코드 세트가 양호한 경우, Route 53은 보조 리소스 레코드 세트의 해당 값으로 DNS 쿼리에 응답합니다.
- 보조 리소스 레코드 세트가 비정상인 경우, Route 53은 기본 리소스 레코드 세트의 상태와 관계없이 기본 리소스 레코드 세트의 해당 값으로 DNS 쿼리에 응답합니다.
- `HealthCheckId` 보조 리소스 레코드 세트의 요소를 생략하고 기본 리소스 레코드 세트가 비정상인 경우, Route 53은 항상 보조 리소스 레코드 세트의 해당 값으로 DNS 쿼리에 응답합니다. 이는 연결된 엔드포인트의 상태에 관계없이 적용됩니다.
  장애 조치 리소스 레코드 세트와 and에 대해 동일한 값을 갖는 비장애 조치 리소스 레코드 세트는 생성할 수 `Name` `Type` 없습니다 .  
  장애 조치 별칭 리소스 레코드 세트의 경우 요소도 포함 `EvaluateTargetHealth` 하고 값을 true로 설정해야 합니다.  
  Route 53의 장애 조치 구성에 대한 자세한 내용은 _Amazon Route 53 개발자 안내서에서 다음 주제를 참조하십시오_.
- Route 53 상태 확인 및 DNS 장애 조치
- 프라이빗 호스팅 영역에서 장애 조치 구성
  *필수 항목 *여부: 아니요  
  _유형_: 문자열  
  _허용되는 값_: `PRIMARY | SECONDARY`  
  _업데이트 필요 사항_: 중단 없음

`GeoLocation`  
 _지리적 위치 리소스 레코드 세트만:_ 쿼리의 지리적 오리진을 기반으로 Amazon Route 53이 DNS 쿼리에 응답하는 방식을 제어할 수 있는 복합 유형입니다. 예를 들어, 아프리카의 모든 쿼리를 IP 주소가 인 웹 서버로 라우팅하려면 `192.0.2.111` a of `Type` 및 a `A` of `ContinentCode` 를 사용하여 리소스 레코드 세트를 생성합니다`AF`.  
프라이빗 호스팅 영역에서 지리적 위치 및 지리적 위치 별칭 리소스 레코드 세트를 생성하는 것은 허용되지만 지원되지는 않습니다.
겹치는 지리적 지역에 대해 별도의 리소스 레코드 세트를 생성하는 경우(예: 한 대륙에 대한 리소스 레코드 세트 하나와 같은 대륙에 있는 국가에 대한 리소스 레코드 세트 하나) 가장 작은 지리적 지역에 우선 순위가 부여됩니다. 이렇게 하면 대륙에 대한 대부분의 쿼리를 하나의 리소스로 라우팅하고 해당 대륙의 국가에 대한 쿼리를 다른 리소스로 라우팅할 수 있습니다.  
동일한 지리적 위치를 지정하는 두 개의 지리적 위치 리소스 레코드 세트를 생성할 수 없습니다.  
요소의 값은 `*` `CountryCode` and 요소에 `Name` 대해 동일한 값을 갖는 다른 지리적 위치 리소스 레코드 세트에 지정되지 않은 모든 지리적 위치와 일치합니다 `Type` .  
지리적 위치는 IP 주소를 위치에 매핑하여 작동합니다. 그러나 일부 IP 주소는 지리적 위치에 매핑되지 않으므로, 7개 대륙을 모두 포괄하는 지리적 위치 리소스 레코드 세트를 생성하더라도 Route 53은 식별할 수 없는 위치에서 일부 DNS 쿼리를 수신합니다. 의 값이 인 리소스 레코드 세트를 생성하는 것이 좋습니다 `CountryCode` `*`. 두 개의 쿼리 그룹, 즉 지리적 위치 리소스 레코드 세트를 생성하지 않은 위치에서 오는 쿼리와 위치에 매핑되지 않은 IP 주소의 쿼리라는 두 개의 쿼리 그룹이 이 레코드에 지정한 리소스로 라우팅됩니다. `*` 리소스 레코드 세트를 생성하지 않으면 Route 53은 해당 위치의 쿼리에 대해 "응답 없음" 응답을 반환합니다.
지리적 위치 리소스 레코드 세트와 and요소에 대해 동일한 값을 갖는 비지리적 위치 리소스 레코드 세트는 생성할 수 `Name` `Type` 없습니다 .  
*필수 항목 *여부: 아니요  
_유형_: GeoLocation  
_업데이트 필요 사항_: 중단 없음

`HealthCheckId`  
상태 확인 상태가 정상일 때만 Amazon Route 53이 DNS 쿼리에 대한 응답으로 이 리소스 레코드 세트를 반환하도록 하려면 요소를 포함 `HealthCheckId` 하고 해당 상태 확인의 ID를 지정합니다.  
Route 53은 다음 중 하나를 기준으로 리소스 레코드 세트가 정상인지 여부를 결정합니다.

- 상태 확인에 지정된 엔드포인트에 주기적으로 요청을 보냅니다
- 지정된 상태 확인 그룹의 상태를 집계하여(계산된 상태 확인)
- CloudWatch 경보의 현재 상태 확인(CloudWatch 지표 상태 확인)
  Route 53은 리소스 레코드 세트에 지정된 엔드포인트(예: 요소의 IP 주소로 지정된 엔드포인트)의 상태를 확인하지 않습니다 `Value` . `HealthCheckId` 리소스 레코드 세트에 요소를 추가할 때, Route 53은 상태 확인에서 지정한 엔드포인트의 상태를 확인합니다.
  자세한 내용은 _Amazon Route 53 개발자 안내서에서 다음 주제를 참조하십시오_.
- Amazon Route 53이 엔드포인트가 정상인지 여부를 판단하는 방법
- Route 53 상태 확인 및 DNS 장애 조치
- 프라이빗 호스팅 영역에서 장애 조치 구성
  **HealthCheckId** 를 지정하는 경우  
  에 대한 값을 지정하는 `HealthCheckId` 것은 Route 53이 DNS 쿼리에 응답하기 위해 둘 이상의 리소스 레코드 세트 중에서 선택하고, Route 53이 상태 확인 상태를 부분적으로 기반으로 선택하기를 원할 때만 유용합니다. 상태 확인 구성은 다음 구성에서만 의미가 있습니다.
- **별칭이 아닌 리소스 레코드 세트**: 라우팅 정책, 이름 및 유형이 동일한 비별칭 리소스 레코드 세트 그룹(예: 이름이 www.example.com A인 여러 가중치 기반 레코드)의 상태를 확인하고 모든 리소스 레코드 세트에 대한 상태 확인 ID를 지정합니다.

리소스 레코드 세트의 상태 확인 상태가 정상인 경우, Route 53은 DNS 쿼리에 응답하는 레코드에 해당 레코드를 포함합니다.

리소스 레코드 세트의 상태 확인 상태가 비정상인 경우, Route 53은 해당 리소스 레코드 세트의 값을 사용하는 DNS 쿼리에 대한 응답을 중지합니다.

그룹 내 모든 리소스 레코드 세트의 상태 확인 상태가 비정상인 경우, Route 53은 그룹 내 모든 리소스 레코드 세트를 정상으로 간주하고 그에 따라 DNS 쿼리에 응답합니다.

- **별칭 리소스 레코드 세트**: 다음 설정을 지정합니다.
- 라우팅 정책, 이름 및 유형이 동일한 리소스 레코드 세트 그룹의 별칭 리소스 레코드 세트(예: 유형이 A인 www.example.com 이름의 여러 가중치 기반 레코드)에 대해 true `EvaluateTargetHealth` 로 설정합니다.
- 별칭 리소스 레코드 세트를 구성하여 동일한 호스팅 영역에 있는 별칭이 아닌 리소스 레코드 세트로 트래픽을 라우팅합니다.
- 별칭이 아닌 리소스 레코드 세트에 대한 상태 확인 ID를 지정합니다.

상태 확인 상태가 정상인 경우, Route 53은 별칭 리소스 레코드 세트를 정상으로 간주하고 DNS 쿼리에 응답하는 레코드에 별칭 레코드를 포함합니다.

상태 확인 상태가 비정상인 경우 Route 53은 별칭 리소스 레코드 세트를 사용하는 DNS 쿼리에 대한 응답을 중지합니다.
**메모**  
별칭 리소스 레코드 세트는 ** 라우팅 정책, 이름 및 유형이 동일한 별칭이 아닌 리소스 레코드 세트 그룹으로 트래픽을 라우팅할 수도 있습니다. 이 구성에서는 상태 확인을 별칭이 아닌 리소스 레코드 세트 그룹의 모든 리소스 레코드 세트와 연결합니다.
**지오로케이션 라우팅\*_  
지리적 위치 리소스 레코드 세트의 경우, 엔드포인트가 비정상이면 Route 53은 더 크고 연결된 지리적 리전에 대한 리소스 레코드 세트를 찾습니다. 예를 들어, 미국의 한 주, 미국 전체, 북아메리카에 대한 리소스 레코드 세트가 있고 모든 위치에 적용되는 `_`is`CountryCode` `\*` 가 있는 리소스 레코드 세트가 있다고 가정해 보겠습니다. 상태 리소스 레코드 세트의 엔드포인트가 양호하지 않은 경우, Route 53은 엔드포인트가 양호한 리소스 레코드 세트를 찾을 때까지 다음 순서로 정상 리소스 레코드 세트를 확인합니다.

- 미국
- 북아메리카
- 기본 리소스 레코드 세트
  **도메인 이름으로** 상태 확인 엔드포인트 지정  
  상태 확인에서 도메인 이름으로만 엔드포인트를 지정하는 경우 각 엔드포인트에 대해 별도의 상태 확인을 생성하는 것이 좋습니다. 예를 들어, `HTTP` 에 대한 콘텐츠를 제공하는 각 서버에 대한 상태 확인을 생성합니다`www.example.com`. 의 값에는 `FullyQualifiedDomainName`리소스 레코드 세트의 이름()이 아닌`us-east-2-www.example.com` 서버의 도메인 이름(예: `www.example.com` )을 지정합니다.  
  상태 점검 결과는 다음을 수행하는 경우 예측할 수 없습니다.
- 리소스 레코드 세트의 이름과 `FullyQualifiedDomainName`동일한 값을 갖는 상태 확인을 생성합니다.
- 해당 상태 확인을 리소스 레코드 세트와 연결합니다.
  *필수 항목 *여부: 아니요  
  _유형_: 문자열  
  _최대_: `64`  
  _업데이트 필요 사항_: 중단 없음

`HostedZoneId`  
레코드를 생성하려는 호스팅 영역의 ID입니다.  
또는 `HostedZoneName` `HostedZoneId`둘 중 하나만 지정합니다. 도메인 이름이 동일한 호스팅 영역이 여러 개 있는 경우 를 사용하여 호스팅 영역을 지정해야 합니다`HostedZoneId`.  
에 `HostedZoneId` 이미 정의되어 있는 경우 `AWS::Route53::RecordSetGroup` 제공하지 마십시오. `HostedZoneId` 둘 다에 정의된 경우 생성이 실패합니다.  
*필수 항목 *여부: 아니요  
_유형_: 문자열  
_최대_: `32`  
_업데이트 필요 사항_: 중단 없음

`HostedZoneName`  
레코드를 생성하려는 호스팅 영역의 이름입니다. 후행 점(예: `www.example.com.`) `HostedZoneName` 을 의 일부로 포함해야 합니다.  
`AWS::Route53::RecordSet` 를 지정하는 `HostedZoneName` 를 사용하여 스택을 생성하면 AWS CloudFormation에서는 이름이 . `HostedZoneName` AWS CloudFormation에서 도메인 이름이 일치하는 호스팅 영역을 찾을 수 없거나 지정된 도메인 이름을 가진 호스팅 영역이 두 개 이상 있는 경우 AWS CloudFormation에서는 스택을 생성하지 않습니다.  
또는 `HostedZoneName` `HostedZoneId`둘 중 하나만 지정합니다. 도메인 이름이 동일한 호스팅 영역이 여러 개 있는 경우 를 사용하여 호스팅 영역을 지정해야 합니다`HostedZoneId`.

`MultiValueAnswer`  
 _다중값 응답 리소스 레코드 세트만_: 트래픽을 웹 서버와 같은 여러 리소스로 대략적으로 무작위로 라우팅하려면 각 리소스에 대해 하나의 다중값 응답 레코드를 만들고 `true` 에 대해 `MultiValueAnswer` 지정합니다. 다음 사항에 유의하세요.

- 상태 확인을 다중값 응답 리소스 레코드 세트와 연결하는 경우, Amazon Route 53은 상태 확인이 정상일 때만 해당 IP 주소로 DNS 쿼리에 응답합니다.
- 상태 확인을 다중값 응답 레코드와 연결하지 않으면 Route 53은 항상 레코드를 정상으로 간주합니다.
- Route 53은 최대 8개의 정상 레코드로 DNS 쿼리에 응답합니다. 정상 레코드가 8개 이하인 경우 Route 53은 모든 정상 레코드로 모든 DNS 쿼리에 응답합니다.
- 정상 상태가 8개 이상인 경우 Route 53은 서로 다른 정상 레코드 조합으로 서로 다른 DNS 해석기에 응답합니다.
- 모든 레코드가 양호하지 않은 경우 Route 53은 최대 8개의 비정상 레코드로 DNS 쿼리에 응답합니다.
- 해석기가 응답을 캐시한 후 리소스를 사용할 수 없게 되면 클라이언트 소프트웨어는 일반적으로 응답의 다른 IP 주소를 시도합니다.
  다중값 응답 별칭 레코드는 만들 수 없습니다.  
  *필수 항목 *여부: 아니요  
  _유형_: Boolean  
  _업데이트 필요 사항_: 중단 없음

`a-z`, `0-9`, 및 `-` (하이픈) 이외의 문자를 지정하는 방법과 다국어 도메인 이름을 지정하는 방법에 대한 자세한 내용은 *Amazon Route 53 개발자 안내서*의 DNS 도메인 이름 형식을 참조하십시오.  
별표(_) 와일드카드를 사용하여 도메인 이름에서 가장 왼쪽에 있는 레이블을 바꿀 수 있습니다(예: `_.example.com`. 다음 사항에 유의하세요.

`Region`  
 _지연 시간 기반 리소스 레코드 세트만:_ 이 리소스 레코드 세트가 참조하는 리소스를 생성한 Amazon EC2 리전입니다. 리소스는 일반적으로 EC2 인스턴스 또는 ELB 로드 밸런서와 같은 AWS 리소스이며, 레코드 유형에 따라 IP 주소 또는 DNS 도메인 이름으로 참조됩니다.  
Amazon Route 53이 지연 시간 리소스 레코드 세트를 생성한 도메인 이름 및 유형에 대한 DNS 쿼리를 수신하면 Route 53은 최종 사용자와 연결된 Amazon EC2 리전 간의 지연 시간이 가장 짧은 지연 시간 리소스 레코드 세트를 선택합니다. 그런 다음 Route 53은 선택한 리소스 레코드 세트와 연결된 값을 반환합니다.

- 모든 Amazon EC2 리전에 대해 지연 시간 리소스 레코드 세트를 생성할 필요는 없습니다. Route 53은 지연 시간 리소스 레코드 세트를 생성하는 리전 중에서 지연 시간이 가장 좋은 리전을 선택합니다.
- 지연 시간이 없는 리소스 레코드 세트는 `Name` 지연 `Type` 시간 리소스 레코드 세트와 and요소에 대해 동일한 값을 가질 수 없습니다.

`Type`  
DNS 레코드 유형입니다. 다양한 레코드 유형 및 데이터 인코딩 방법에 대한 자세한 내용은 *Amazon Route 53 개발자 안내서*의 지원되는 DNS 리소스 레코드 유형을 참조하십시오.  
기본 리소스 레코드 세트에 유효한 값: `A` | `AAAA` | `CAA` | `CNAME` | `DS` |`MX` | `NAPTR` | `NS` | `PTR` | `SOA` | `SPF` | `SRV` | `TXT`  
가중치 적용, 지연 시간, 지리적 위치 및 장애 조치 리소스 레코드 세트에 대한 값: `A` | `AAAA` | `CAA` | `CNAME` | `MX` | `NAPTR` | `PTR` | `SPF` | `SRV` | `TXT`을 클릭합니다. 가중치 기반, 지연 시간, 지리적 위치 또는 장애 조치 리소스 레코드 세트의 그룹을 생성할 때 그룹의 모든 리소스 레코드 세트에 대해 동일한 값을 지정합니다.  
다중값 응답 리소스 레코드 집합에 유효한 값: `A` | `AAAA` | `MX` | `NAPTR` | `PTR` | `SPF` | `SRV` | `TXT`  
SPF 레코드는 이전에 이메일 메시지를 보낸 사람의 ID를 확인하는 데 사용되었습니다. 그러나 의 값이 인 리소스 레코드 세트를 만드는 것은 더 이상 권장되지 않습니다 `Type` `SPF`. RFC 7208, *SPF(Sender Policy Framework) for Authorizing Use of Domains in Email, 버전 1*이 "...[나는]에 정의된 ts 존재 및 메커니즘 [RFC4408] 으로 인해 몇 가지 상호 운용성 문제가 발생했습니다. 따라서 SPF 버전 1에는 더 이상 적합하지 않습니다. 구현은 그것을 사용하지 않습니다." RFC 7208에서 섹션 14.1, SPF DNS 레코드 유형을 참조하십시오.
별칭 리소스 레코드 세트의 값:

- **CloudFront 배포:** `A`

배포에 대해 IPv6가 활성화된 경우, 트래픽을 배포로 라우팅하기 위해 두 개의 리소스 레코드 세트(값이 있는 레코드 세트 와 값이 있는 레코드 세트 `A` )를 생성합니다`AAAA`.

- **리전화된 하위 도메인이 있는 Amazon API Gateway 환경**: `A`
- **ELB 로드 밸런서:** `A` | `AAAA`
- **Amazon S3 버킷:** `A`
- **Amazon Virtual Private Cloud 인터페이스 VPC 엔드포인트** `A`
- **이 호스팅 영역의 다른 리소스 레코드 세트:** 별칭을 생성할 리소스 레코드 세트의 유형을 지정합니다. 및 를 제외한 모든 값이 지원됩니다 `NS` `SOA`.
  **메모**  
  호스팅 영역과 이름이 같은 별칭 레코드(zone apex라고 함)를 생성하는 경우 값이 인 레코드로 트래픽을 라우팅할 수 없습니다 `Type` `CNAME`. 이는 별칭 레코드가 트래픽을 라우팅하는 레코드와 동일한 유형이어야 하며, 별칭 레코드에 대해서도 zone apex에 대한 CNAME 레코드 생성이 지원되지 않기 때문입니다.
  _필수 항목 여부_: 예  
  _유형_: 문자열  
  _허용되는 값_: `A | AAAA | CAA | CNAME | DS | MX | NAPTR | NS | PTR | SOA | SPF | SRV | TXT`  
  _업데이트 필요 사항_: 중단 없음

`Weight`  
 _가중치 기반 리소스 레코드 세트만:_ DNS 이름과 유형의 조합이 동일한 리소스 레코드 세트 중에서, Amazon Route 53이 현재 리소스 레코드 세트를 사용하여 응답하는 DNS 쿼리의 비율을 결정하는 값입니다. Route 53은 DNS 이름과 유형의 조합이 동일한 리소스 레코드 세트에 대한 가중치의 합계를 계산합니다. 그런 다음 Route 53은 총계에 대한 리소스의 가중치 비율에 따라 쿼리에 응답합니다. 다음 사항에 유의하세요.

- 모든 가중치 기반 리소스 레코드 세트의 요소`Weight` 값을 지정해야 합니다.
- 가중치 기반 리소스 레코드 세트당`ResourceRecord` 하나만 지정할 수 있습니다.
- 지연 시간, 장애 조치 또는 지리적 위치 리소스 레코드 세트는 `Name` 가중치 기반 리소스 레코드 세트와`Type` and 에 대해 동일한 값을 가질 수 없습니다.
- 및 요소에 `Name`대해 동일한 값을 갖는 가중치 기반 리소스 레코드 세트를 최대 100개까지 생성할 수 `Type` 있습니다.
- 가중치 기반(가중치 기반 별칭 아님) 리소스 레코드 세트의 경우, `Weight` `0` 리소스 레코드 세트에 대해 로 설정하면 Route 53은 해당 리소스 레코드 세트에 적용 가능한 값을 가진 쿼리에 응답하지 않습니다. 그러나 `Weight` `0` DNS 이름과 유형의 조합이 동일한 모든 리소스 레코드 세트에 대해 로 설정하면 트래픽이 동일한 확률로 모든 리소스로 라우팅됩니다.

  to로 설정하면 `Weight` `0` 상태 확인을 가중치 기반 리소스 레코드 세트와 연결할 때 효과가 달라집니다. 자세한 내용은 Amazon Route 53 개발자 안내서의 Route 53 액티브-액티브 및 액티브-패시브 장애 조치 구성 옵션을 참조하십시오\**.
